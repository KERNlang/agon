# Manual Axis: Polyglot Runtime Bridge

> **Not generated by `kern self-coverage` — hand-maintained. Preserve on regeneration.**
> The tables above measure single-target (TS) migratability: "this handler *could* be native KERN but the TS-target grammar can't express it yet." The gaps below are a different class — they track a **new emit axis** (a Python target + a cross-runtime bridge), not handlers blocked from lifting into KERN within the existing TS target.

## Why this is a different kind of gap

KERN today emits one target (TS) per file. A whole class of agon code is *intentionally not KERN* because it must run on a different runtime: `kern_engines/` drives engine TUIs with OS-level pty primitives (`forkpty`, winsize ioctls, fd reads) that have no good TS equivalent and exist as stdlib only in CPython. Closing this axis lets KERN author that code too — and, at the endgame, run it.

## Proof-of-concept — the manual bridge that defines the spec

`kern_engines/` is the hand-written reference for what a generated KERN polyglot bridge must produce:

| File | Lines | Role | KERN's job |
|---|---|---|---|
| `kern_engines/cli/session.ts` | 379 | TS side: subprocess spawn, NDJSON correlation, streaming waiter, timeout/error re-inflation | **Generate** — pure seam |
| `kern_engines/cli/daemon.py` | 167 | Python side: NDJSON event loop, dispatch `ask`/`ask_stream`/`close` | **Generate** — pure seam |
| `kern_engines/cli/pty_session.py` | 875 | OS-level pty driver (forkpty, winsize, fd reads, ANSI strip, response extraction) | **Stay Python** — author in KERN, run on CPython |
| `kern_engines/cli/configs.py` | 143 | Per-engine empirical config (markers, `chrome_regex`, paste tuning) | **Stay Python** — data |

The **seam** (546 lines, `session.ts` + `daemon.py`) is what KERN should generate from one cross-call declaration. The **driver + config** stay Python. The goal is "never hand-write the seam," **not** "transpile the driver."

## Endgame: KERN runtime (beyond the subprocess bridge)

Stages 1-7 generate the *subprocess* seam. The endgame removes the seam: a **KERN runtime** that embeds CPython + a JS engine in one host process (precedent: GraalVM/Truffle, PyO3, CPython's C-embedding API), turning each cross-runtime call into an in-process call over a shared heap. `os.forkpty` works because a real CPython engine lives *inside* the runtime — the boundary is permanent only on stock Node, not on a polyglot runtime.

Two hard problems gate the runtime:
- **Native-extension ABI** — embed real CPython for full compat (numpy C core, `termios`, native npm addons); any reimplementation chases native compat forever.
- **GIL vs event loop** — one process, two concurrency models (CPython GIL vs libuv async): who owns the main thread, how a JS `await` interleaves with a blocking Python call. The core systems work.

**Build sequencing:** subprocess bridge (today, hand-written) → KERN-generated bridge (stages 1-7) → embedded in-process runtime. Each step peels off a layer of seam.

**Note on motivation:** agon's per-call cost is seconds-long LLM dispatch, so the *bridge* (stages 1-7) is sufficient for agon's needs. The *runtime* is motivated by the language vision and by big-in-memory-object use cases (tensors/dataframes, where serializing across a pipe is the bottleneck) — not by agon's performance.
